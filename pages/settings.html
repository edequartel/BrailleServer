Connection

Configure the localhost endpoints used for BrailleBridge HTTP calls and WebSocket key events.

Local API

Used to write text to the braille display and clear output.

Example endpoints: /braille, /clear, /status

WebSocket

Used to receive generalized key events from the braille display.

Typical usage: subscribe to key events and map them to actions.

Auto connect

Behavior when the page loads.

Connect on startup
Attempt to connect to WebSocket and probe API status.
Auto reconnect
Retry after disconnect with exponential backoff.

Bridge profile

Select how strictly BrailleServer should follow the bridge capabilities.

Safe mode disables advanced formatting and reduces API calls.

Braille display

Settings for braille output, cursor behavior, and optional on-screen braille monitor.

Output format

How text should be rendered on the physical display.

Show cursor
Display a cursor cell in the output (if supported).

Braille monitor

Optional visual mirror of braille cells for sighted support staff.

Enable monitor
Show the monitor component on activity pages.
This is only UI; it does not affect the physical display.

Audio

Configure audio playback, voice settings, and latency mitigation for short prompts.

Playback

How the UI plays instruction audio and effects.

Preload audio
Preload next audio file(s) to reduce gaps.
Play short "lead-in"
Optional micro-sound before TTS/streaming audio.

Voice

Mock voice selection (for TTS engines like ElevenLabs or local TTS).

For large instruction sets, MP3 files reduce runtime cost and latency.

Content

Pick language packs and base URLs for audio and JSON content.

Language

Select the primary content language.

This selects the language folder paths in sounds.json (mock).

Base URL

Where audio and JSON are served from (GitHub Pages, server, etc.).

If you change this, ensure CORS and correct paths for MP3/JSON.

Privacy

Student progress can be stored locally in the browser without sending data to a server.

Local progress

Mock settings for local-only tracking.

Enable local progress tracking
Stores progress in LocalStorage/IndexedDB on this device.
Anonymous mode
Do not store student identifiers; store only activity completion.

Reset

Clear all local settings and progress (mock action).

This would clear LocalStorage/IndexedDB for this origin.

Advanced

Developer options: logging verbosity, diagnostics overlays, and feature flags.

Logging

Mock logging levels for browser console and on-page event log.

Developer mode should never be default for student use.

Feature flags

Mock feature toggles used during refactors.

Use new activity controller
Switch between legacy and unified activity flow.
Strict JSON schema
Reject content that does not validate.
Mock page only. No data is persisted. Intended as a UI template for your repo.
